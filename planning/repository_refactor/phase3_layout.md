# Phase 3 — ディレクトリ構成と依存ルール

[全体計画](overall_plan.md) の Phase 3。Phase 1 の能力グループと Phase 2 の設計方針を、**`src/` 以下の配置案**と**検証可能な依存ルール**、**Phase 4 の縦スライス順**に落とす。具体的な型名・イベント列挙の確定は Phase 4 着手時でよいが、モジュール境界の「上下関係」は本書で固定する。

**実施記録**: 2026-05-09 — 初版作成（提案ツリー、依存許可表、縦スライス順、Phase 1 対応表）。

## 目的

- Phase 1 の各能力グループが、提案ツリー上の **いずれかの葉／サブツリーに一意に対応付け可能**であることを示す。
- Phase 2 の「コアは Bevy 非依存」「I/O はポート背後」「UI は薄いアダプタ」を、**許可依存の表**として検証できる形にする。
- Phase 4 での移行を **縦スライス**に分解し、big-bang を避ける順序を列挙する。

## 計画概要

### Task 1 — Phase 1 グループ → 配置領域の対応表

- [全体計画](overall_plan.md) の Phase 3 受け入れ「各グループに対応付け可能」を満たすため、下文「提案ディレクトリ構成」と「能力グループ対応表」を埋める。
- 現行のフラットな `src/*.rs` は参照しつつ、**移行先の論理領域名**だけを書く（ファイル単位の機械的マッピング表は Phase 4 の PR 単位でよい）。
- [ ] 機能一覧の各大分類（フラクタルモデル・Seed・Placement・Undo・プリセット・パラメータ・Result・ビュー入力・レイアウト・共有・エクスポート・プラットフォーム・フィードバック・制約・予算）が、対応表のいずれかの行に載っている。

### Task 2 — 提案ディレクトリ構成（論理ツリー）

- Phase 2 の「機能ドメイン優先」「四層ディレクトリ名の硬性禁止」に沿い、**実装クレート直下**の候補ツリーを一文説明付きで固定する。
- 名前は確定案とし、Phase 4 で長さ・可読性のために微調整してよいが、**依存の向き**は変えない。
- [ ] ツリーに「エントリ（プラグイン登録・ターゲット別初期化）」「コア（Bevy 非依存）」「アプリケーション状態・ユースケース」「プレゼンテーション（egui／キャンバス入力）」「インフラ（I/O・環境 API）」の役割が読み分けられる。

### Task 3 — モジュール間依存ルール（検証可能）

- 「モジュール A は B に依存してよい／よくない」を **行列または一覧**で書く。将来 Phase 4 末尾で grep／レビュー観点にできる粒度にする。
- **コア → 上位への依存禁止**、**インフラが UI に依存しない**、**プレゼンテーションが共有形式の詳細ロジックだけを抱えない** 等、Phase 2 の一文方針と矛盾しないことを明示する。
- [ ] ルールを一文で破っている依存経路がないか、本文を読んだだけで検証できる。

### Task 4 — ポート（trait）の置き場

- アプリケーションがインフラ具象を見ないための **ポート**を、`app` 内サブモジュールと `platform` のどちら側に置くかを決める（推奨：**trait はアプリケーション側または共通の薄い `ports` 領域**、実装のみ `platform`）。
- [ ] クリップボード・ファイル保存・URL／履歴・Web Share など Phase 1「インフラ」に相当する能力が、すべてポート経由の説明に収まっている。

### Task 5 — 移行方針と縦スライス順

- **推奨：縦スライス**（機能単位で旧モジュールから新配置へ移し、各スライス後にビルド・スモーク）。**一括 big-bang は非推奨**。
- 既存の公開シンボルを壊さないための **一時ファサード**（旧 `mod` を新配置へリエクスポートする期間）を許容する旨を書く。
- [ ] 縦スライスの **推奨順序**が本文に列挙されている（少なくとも 4 ステップ以上の粒度）。

## 計画詳細

- **エントリポイント**：バイナリが単一クレートのままでも、`main` はプラグイン登録とターゲット別ブートストラップに薄く留め、論理本体はサブツリーへ寄せる方針とする（詳細は Phase 4）。
- **プリセット**：データと「適用」手続きが分離しうる。データはコア寄りの静的结构、適用と Undo 境界はアプリケーション側の説明に載せる。
- **共有とエクスポート**：符号化・ポリシーはアプリケーション、バイト列・ダイアログ・ブラウザ API はインフラ、進捗・結果メッセージの見せ方はプレゼンテーション、という Phase 2 の分割を対応表で崩さない。
- **`fractal_presets` のような既存ディレクトリ**：Phase 4 で「コアデータ＋一覧」へ寄せるか「アプリケーションのプリセット適用」近傍へ寄せるか選択するが、Phase 3 では **論理領域名**だけを固定する。

## 提案ディレクトリ構成（候補ツリー）

※ ディレクトリ名は識別子レベルの案。コード・型定義は書かない。

```
src/
  core/           … フラクタル・種図形・IFS・再帰結果などの純データ・純計算（Bevy 非依存）
  app/            … ワークスペース状態、Undo／Redo 境界、プリセット適用、共有符号化／復号、エクスポート駆動とエラー種別（イベント・システムで編成）
  ports/          … アプリケーションが定義する I/O 抽象のみ（オプション：`app/ports` に統合してもよい）
  ui/             … egui パネル、レイアウト、キャンバス入力の解釈、結果ビュー操作、トースト表示
  platform/       … クリップボード、URL／履歴、ファイルダイアログ、WASM 初期化・復元、共有 PNG の受け渡しなど具象実装
  bootstrap/      … プラグイン登録順・レンダーレイヤ等の「束ね」（省略時は main 側に薄く残す）
  main.rs         … エントリ（Phase 4 で bootstrap へ寄せるかは実装時判断）
```

**読み替えルール**：上記が過剰に細かいと判断した場合、`ports` を `app` に折りたたむ、`bootstrap` を `main` に残す、といった **flatten は可** とする。ただし **`core` を Bevy 依存にしない**ことと、**`platform` が `ui` に依存しない**ことは維持する。

## モジュール間依存ルール（許可関係）

検証のしかたの例：`use` による依存が、許可されているセルに無ければ違反とみなす。

| 依存元 → 依存先 | `core` | `ports` | `app` | `ui` | `platform` | `bootstrap` |
|-----------------|--------|---------|-------|------|------------|-------------|
| `core` | （内部のみ） | × | × | × | × | × |
| `ports` | ○（境界型のみに限定） | （内部のみ） | × | × | × | × |
| `app` | ○ | ○ | （内部のみ） | × | ×（具象クレート参照禁止；trait 実装は別クレートまたは `platform` であり `app` は宣言のみ） | × |
| `ui` | ○（表示用に読むのみ） | × | ○（イベント・読み取り） | （内部のみ） | × | × |
| `platform` | ○（シリアル／バイト列境界で必要なら） | ○（trait 実装） | × | × | （内部のみ） | × |
| `bootstrap` | × | × | ○ | ○ | ○ | （内部のみ） |

**補足（同等なら違反ではない）**

- `ports` を `app` に統合した場合、表の `ports` 列は `app` の「公開 trait 部分」に読み替える。**ただし** `platform` がその trait を実装するには、`trait` の宣言が `platform` より上位の共有モジュール（独立した `ports` など）にあり、**`app` と `platform` が相互に `use` しない**構成にすることが前提となる（単一クレート内ではサブモジュール順と `use super` の整理でこれを満たす）。
- `bootstrap` から `core` へ直接依存させないのは、起動処理がドメイン計算を肥大化させないための推奨。初期状態組み立てにドメイン Constructor が必要な場合のみ **最小限** の依存を Phase 4 で許可し、本表に脚注で追記する。

## Phase 4 向け：縦スライス（推奨順序）

1. **`core` の切り出し** — IFS・種図形・再帰結果など Bevy 非依存部分を移動し、既存モジュールから再エクスポートするファサードでビルドを維持する。
2. **ポートと `platform` の輪郭** — クリップボード・URL・ファイル・Web Share 等を trait に集約し、既存実装を `platform` に寄せる（挙動変更なしを最優先）。
3. **`app` への状態・ユースケースの集約** — 共有符号化、Undo、プリセット適用、深度クリップ・予算などを一段にまとめ、`ui` からはイベント経由に統一していく。
4. **`ui` の分割整理** — パネル・ビューポート・トーストをサブモジュール化し、`app` への依存を読み取り専用＋イベントに限定する。
5. **`bootstrap`／`main` の薄型化** — プラグイン登録とシステム順序をここに閉じ、ファサードを削除して import を新ツリーへ寄せる。

各スライスの後：**`cargo build`（およびプロジェクト標準の WASM ビルドがあればそれも）**と、起動・編集・共有・エクスポートのスモークを実施する（詳細チェックリストは [Phase 4 — 実装移行](phase4_implementation.md)）。
