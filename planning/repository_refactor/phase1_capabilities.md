# Phase 1 — 機能洗い出し・責務整理

[全体計画](overall_plan.md) の Phase 1。現状の `src` 構成は参照せず、提供能力とその負う責務のラベルを固定する。

**実施記録**: 2026-05-09 — 入力経路・共有・エクスポート・編集モジュールを横断して一覧を追補済み。2026-05-09 — 共有 PNG 保存とトーストの連携（同期／非同期経路・ネイティブ保存の成否メッセージ）を再確認して本文を整合。

## 目的

- ユーザーに見える能力と、アプリが暗黙に担う責務（プラットフォーム差・安全制限など）を漏れなく列挙する。
- 各項目に **一貫した観点** で責務タグを付与し、Phase 2（パターン）・Phase 3（配置）で参照できる「能力カタログ」にする。
- Phase 2 着手時に「別の名前のグループ」を再発明しないための共通語彙を揃える。

## 計画概要

### Task 1 — 能力の単票化

- 下記「機能一覧」をレビューのたたきにする。
- 追う対象：**メニューや画面上の操作**、**状態が URL などに載るか**、**起動〜終了までのプラットフォーム特有挙動**。
- [x] 画面上の入力経路を一通り確認（マウス／キーボード／タッチ、主要ボタン・メニュー・ショートカット）。
- [x] 「開発者向けのみ」の能力を **（開発・検証向け）** で明示。

### Task 2 — ラベル付けルールの固定

- 各項目に次の **いずれか主タグ** を付ける（副タグは備考欄に短文で最大 1 つのみ）。
  - **コアドメイン**: フラクタルが何か（基図形・IFS・深さ・座標モデル）に関する不変に近い定義。
  - **アプリケーション**: ユースケースの編成（Undo 対象の区切り、プリセット適用、共有の読み書きの手順など）。
  - **プレゼンテーション**: レイアウト、egui パネル、入力の解釈を UI に近い層で扱う部分。
  - **インフラ**: ファイル・URL・履歴・WASM API・レンダーパイプライン等の外部境界。
- [x] 迷いどころは「ユーザー表面」か「交換可能な手段」で主タグを決定済み。

### Task 3 — ギャップ・重複の棚卸し

- 同じ能力が別の言い方で二重に書かれていないか確認する。
- 「能力はあるがユーザーから直接は触れない」制約は **制約・予算** の表に集約する。
- [x] プレースホルダだけのグループはない。

### Task 4 — 文書の確定と全体計画との接続

- [x] 下文のチェックリストと [全体計画](overall_plan.md) Phase 1 受け入れ条件を照合済み。
