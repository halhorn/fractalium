# Phase 2 — パターンとアーキテクチャ方針

[全体計画](overall_plan.md) の Phase 2。Phase 1 の能力カタログを前提に、**Bevy／egui／ネイティブ・WASM** に整合する設計の言い方を固定する。モジュール名・型名・具体的なイベント一覧は Phase 3 以降で決める。

**実施記録**: 2026-05-09 — 初版および設計方針ドラフト作成。2026-05-09 — WASM／ネイティブ両対応を当面維持する旨を確定し、Phase 2 受け入れを満たすとして完了扱い。

## 目的

- ECS・システム・UI の責務の切り方を短文でそろえ、Phase 3 のディレクトリ案と依存ルールが空振りしないようにする。
- Phase 1 の **主タグ**（コアドメイン／アプリケーション／プレゼンテーション／インフラ）と Bevy の概念の対応原則を一文レベルで明示する。
- 「採用する／採用しない」を理由つきで残し、Phase 4 で過剰抽象化や矛盾した依存を避ける。

## 計画詳細

- 具象の **ディレクトリツリー・型名・public API** は Phase 3 で決める。本書は「なぜそう分割するか」の前提のみを与える。
- 一般的なクリーンアーキテクチャの四層ラベルを **そのまま** ディレクトリ名に貼らない方針を、採用しないパターンと整合させて書く。
- イベントやリソースの **網羅リスト** は書かず、「どの種の変化がどの責務に属するか」の原則に留める。

## 設計方針（Bevy／egui／マルチターゲット）

※ コード・型名なし。Phase 4 でコードに落とすときの合言葉として使う。

### Phase 1 主タグの写像（原則）

| Phase 1 主タグ | Bevy／周辺での主な表現 | 備考 |
|----------------|------------------------|------|
| コアドメイン | 純粋なデータ・不変に近い計算を担う型群。World の **リソース** として保持するか、**特定のシステム集合だけ**が読み書きするリソースに閉じ込める。 | アプリ状態の唯一の論理モデルとして扱う。 |
| アプリケーション | ユースケースの編成（Undo 境界、プリセット適用、共有符号化／復号、書き出しパイプの駆動・エラー種別など）。**システム** と **イベント**（コマンド的な要求）で表す。 | UI は「何をさせるか」をイベントやメソッド呼びに縮約し、ルール本体はここに置く。 |
| プレゼンテーション | egui のパネル、レイアウト、入力のヒット検査、結果ビュー上の見た目と操作。**システム**として毎フレーム UI を更新し、意図を **イベント** に変換するだけに留める。 | フラクタル定義そのものや共有形式の詳細ロジックを持たない。 |
| インフラ | OS／ブラウザ API、ファイルダイアログ、クリップボード、URL と履歴、WASM の起動復元。**trait オブジェクトまたはターゲット別モジュール**の背後へ集約する。 | Bevy の `Resource` に「実装を差し替え可能なポート」として載せ、アプリケーション層からは具象を見ない。 |

### 状態の単一ソース

- ワークスペースの編集結果（種図形・レプリカ・深さなどを含む）の **論理上のソース** は一系統とし、UI はそのスナップショットを表示・入力で更新要求するのみとする。
- Undo／Redo は **離散的編集単位** のスタックであり、その境界の定義はアプリケーション側の規則とする（Phase 1 の「連続スライダーは対象外」と整合）。
- 「共有用クエリ」「表示用キャメラ」など **派生物** は、ソースから導けるものとして明示的にどの層で再計算／同期するかを Phase 3 で配置に落とす（本書では「ソースとキャッシュを二重管理しない」を原則とする）。

### システム境界（ECS）

- システムの責務は **短く**：入力→イベント、イベント→状態、状態→メッシュや描画用データ、UI 同期、インフラ I/O の起動などに分割する。
- 同一フレーム内の順序に依存する処理は **明示的な段階**（プラグイン内のセットアップ順、または同一概念のグルーピング）で束ねる方針とし、「どこでも早い順に読み書きする」状態を避ける。
- Bevy に強く結びつくのはプレゼンテーションとインフラの **適合（adapter）部分** とし、コアドメインの数式・IFS 組み立てなどは ECS の詳細から独立した **純関数・純データ** に近い形を維持する。

### egui と Bevy の接続

- egui は **薄い適合層**：ユーザーのクリック／ドラッグ／ショートカットは、可能なら **意図**（例：複製選択、種図形の一端移動開始）へ解釈してからイベント化する。
- 種図形・IFS・共有ペイロードなど **コアの不変条件** を、プレゼンテーションから直接書き換えず、アプリケーション側のルールとイベント処理に寄せる。
- 長時間ブロックする処理や環境側のダイアログは **インフラ port** に委ね、完了結果は別経路で **フィードバック用のリソースまたはイベント** に載せ、トースト等で表示する（Phase 1「同期／非同期のトースト」と整合）。
- メニューや結果パネルの **開閉状態** が書き出しや共有のトリガとなる場合、その連携規則はアプリケーション側のポリシーとして書くこと（詳細は Phase 3）。

### インフラ（I／O・プラットフォーム）

- **同じユーザー能力** に対して、ネイティブと WASM で API が異なるときは、アプリケーションは **共通のポート**（操作の開始／完了／失敗理由）だけを見る。
- **`cfg`** は「そのターゲットでコンパイルされないコード」を隔離するために使い、**振る舞いの分岐そのもの**は可能なら trait で表す（両方載るバイナリでのテストや将来の構成変更に備える）。
- URL フラグメントの読み書き、共有 PNG のバイト列の受け渡しなどはインフラの責務とし、アプリケーションは「符号化済み」「バイト列」など **境界で型付けされた単位** だけを扱う。
- 環境側の **非同期完了**（ファイル保存ダイアログ、Web Share など）は、**メインの Bevy 更新ループ**をブロックしない。結果はポート経由で **リソースまたはイベントに載せ、後続フレームで処理する**（中央のシステムで長時間ブロッキングしない）。

### 過剰抽象化の自制

- 将来の機能のための「汎用フレームワーク層」を Phase 4 で先に広げない。必要になったときにのみ trait またはモジュールを追加する。
- Phase 3 で依存ルールに落とせないほど複雑な中間層は作らない。迷ったら縦スライス（機能単位移行）優先と [全体計画](overall_plan.md) で揃える。

### 制約・予算（Phase 1 との関係）

- 深さ上限・線数上限・書き出し解像度上限などは **アプリケーション規則**（検証・スライダー上限）として実装し、インフラは「拒否」「容量超過」を返せれば足りる。表示文言はプレゼンテーションに近いシステムで扱ってよい。

### 方針（確定）

- **WASM／ネイティブの両対応は当面維持する**（運用上メインがブラウザでも、ネイティブ向けビルドは当面スコープから外さない）。単一ターゲットへの縮退を検討する場合は、本書と [全体計画](overall_plan.md) をあらためて更新する。

## 意図的に採用しないパターン

| パターン | 理由（概要） |
|----------|----------------|
| クリーンアーキテクチャの四層を **ディレクトリ名と一対一** で強制する | Bevy は「システム＋リソース」の実行モデルが主であり、レイヤ名だけではシステム順序やイベントとの関係が隠れる。フェーズ別の名前は Phase 3 で **機能ドメイン** ベースを優先する。 |
| コアドメインが **`bevy` クレートに依存**する | IFS と幾何の意味を単体検証や将来の再利用から切り離せなくなる。数値・アルゴリズムは ECS から独立したモジュール群に閉じる。 |
| **巨大単一プラグイン** に全システムを詰める | 境界が読めず Phase 4 のレビューも困難。ドメインごとにプラグインまたはモジュール塊へ分ける前提とする（配置は Phase 3）。 |
| UI から直接ファイルパス／URL 文字列へ **常時書き込む** | テスト可能性とプラットフォーム差の分散を招く。I／O と文字列運搬はインフラ port に集約する。 |
| グローバルなシングルトンや **静的ミュータブル** でアプリ状態を保持する | World 内リソースとの二重ソースになり、確定順序の更新が崩れやすい。 |
