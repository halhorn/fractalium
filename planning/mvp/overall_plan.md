# MVP 計画

## 目的

「基本図形を描き、複製ルールを与えると、フラクタルが描かれる」最小体験を成立させる。これにより、ユーザーが本プロダクトの中核体験を確認でき、以降の Phase の方向性を判断できる状態にする。

## 計画概要

MVP は **Foundation（基盤）** と **Features（機能）** の 2 段で構成する。Foundation を先に着手し、Features は Feat1 → Feat2 → Feat3 → Feat4 の順で進めるのが基本（依存順）。各項目の詳細は着手直前に対応するファイル / ディレクトリへ追記する。

### Foundation（基盤）

| ID | 概要 |
|----|------|
| F1 | プロジェクト初期化（ビルドツール・型・lint・整形・テスト・README） |
| F2 | アプリ骨組み（操作部と結果表示部のレイアウト・ルーティング不要の単一ページ） |
| F3 | 正規化座標キャンバスコンポーネント（[-1, 1] × [-1, 1] 正方領域、画面サイズ追従） |
| F4 | フラクタル状態モデル（基本図形・複製ルール・深さ）と状態管理層 |

### Features（フィーチャー）

各 Feature は `mvp/features/<feature_name>/` 配下にディレクトリを切り、必要なら詳細計画を追加する（着手直前で良い）。

| ID | 概要 |
|----|------|
| Feat1 | 基本図形の描画: ドラッグで直線を引け、複数の直線が基本図形として保持される |
| Feat2 | 複製の配置: 位置・角度・スケールを持つ複製を 1 つ以上配置・編集・削除できる |
| Feat3 | フラクタル描画: 結果表示部で深さ N まで再帰的に変換を適用して描画する |
| Feat4 | リアルタイム反映: 操作部の変更が結果表示部へ即時に反映される |

## 受け入れ条件

### Foundation

- [ ] F1 完了: `npm run dev` 等で開発サーバが起動し、空ページが表示される / lint と型チェックが CI もしくはローカルスクリプトで通る
- [ ] F2 完了: 操作部と結果表示部の 2 領域が画面にレイアウトされている（中身は空でよい）
- [ ] F3 完了: [-1, 1] 座標系で点・線を指定すると、キャンバスの正しい位置に描画される（ウィンドウリサイズで歪まない）
- [ ] F4 完了: 基本図形・複製・深さを保持する状態モデルが定義され、状態変更が UI に伝搬する

### Features

- [ ] Feat1 完了: 操作部キャンバス上をドラッグして直線を引ける / 複数本引ける / 引いた線が状態として保持される
- [ ] Feat2 完了: 複製を 1 つ以上追加できる / 各複製の位置・角度・スケールを数値入力もしくはドラッグで編集できる / 複製を削除できる
- [ ] Feat3 完了: 結果表示部に少なくとも深さ N=4 までのフラクタルが描画される / 深さを UI から変更できる
- [ ] Feat4 完了: 操作部での変更が結果表示部に 1 秒以内（体感的に即時）に反映される

### 動作確認

- [ ] 既知のフラクタル例が再現できる（例: 1 本の直線を基本図形とし、左右 2 つの複製で「Y 字」を作ると、深さを上げると樹木状フラクタルになる）
- [ ] README に起動方法と簡単な使い方が記述されている

## 範囲外（MVP では実装しない）

- 図形プリセット（四角形・三角形など）→ Phase 2
- 複製パラメータの同期・グルーピング → Phase 3
- 保存・読込・エクスポート（PNG/SVG）→ Phase 4
- undo/redo・スナップ・ガイド等の高度な編集 UI → Phase 4
- 色・線幅などのスタイル詳細編集（最低限の固定スタイルのみ採用）

## 補足

### 操作部 UI 概形（暫定）

- 中央: 基本図形を描く [-1, 1] キャンバス（複製の配置プレビューも同キャンバス上に薄く重ねる想定）
- サイドパネル: 複製のリスト・追加削除・各複製のパラメータ編集
- ヘッダ or サイド: 描画深さスライダ

[Question]
操作部のキャンバスと、複製配置編集のキャンバスを統合するか分離するか。MVP ではどちらを採るか。
[/Question]

[Answer]
統合案を採用する: 1 つのキャンバス上に基本図形（実線）と各複製の輪郭（半透明枠）を重ねて表示する。複製の輪郭は基本図形のバウンディングボックスを変換して描画。これにより画面構成がシンプルになり、操作の対応関係も直感的。
[/Answer]

### 結果表示部の描画方式

[Question]
深さ N での再帰描画を毎フレームやり直すか、深さ別にキャッシュするか。MVP の性能要件は？
[/Question]

[Answer]
MVP では毎回再描画する素朴実装で十分とする。深さは小さい範囲（例: 最大 7 程度）に制限し、操作変更時のみ再計算する。深い階層・大量複製での最適化は Phase 4。
[/Answer]
