# Phase 4 — 実装移行

[全体計画](overall_plan.md) の Phase 4。[Phase 3 — ディレクトリ構成と依存ルール](phase3_layout.md) で固定したツリー・依存許可表・縦スライス順に従い、**挙動を維持したまま** `src/` を再編する。型名・ファイル名の機械的マッピング表は本書では持たず、各スライス着手時にコードと diff で確定する。

**実施記録**: （着手時に日付とマイルストーンを追記）

## 目的

- Phase 3 の論理配置と依存の向きを、実行可能なクレート構造へ反映する。
- big-bang を避け、縦スライスごとにビルドと主要動作を検証し、リグレッションリスクを抑える。
- 移行完了時に、Phase 3 の依存ルール違反が残っていない状態にする（レビューまたは簡易検証で確認可能）。

## 計画概要

### 前提

- [Phase 3](phase3_layout.md) の受け入れ（対応表・依存表・縦スライス）が満たされている、または不足分を先に本書と整合させてから着手する。
- [Phase 2 — パターンとアーキテクチャ方針](phase2_patterns.md) の「状態の単一ソース」「I/O はポート背後」「UI は薄いアダプタ」を、各スライスの完了定義に含める。

### Task 1 — `core` の切り出し

- Bevy 非依存の純データ・純計算を `core`（または Phase 3 で確定した同等領域）へ移す。
- 既存の公開パスを壊さないよう、**一定期間ファサード**（旧モジュールからの再エクスポート）でビルドを維持する。
- 本スライス完了時：**ネイティブ** `cargo build` が成功する。
- [ ] `core` が Bevy・egui・プラットフォーム具象に依存していない（許可表どおり）。

### Task 2 — ポートと `platform` の輪郭

- クリップボード・URL／履歴・ファイル・Web Share 等をアプリケーションが参照する **抽象（ポート）** に集約し、具象は `platform` 側へ寄せる。
- `cfg` 分岐と trait 実装の置き場は Phase 2／3 の方針に合わせ、**厚い分岐の散在**を増やさない。
- 本スライス完了時：ネイティブ `cargo build` に加え、**WASM** は CI と同旨の `trunk build --release`（リポジトリ標準の引数があればそれ）が成功する。
- [ ] `app` が `platform` の具象に直接依存していない（ポート経由の説明で足りる）。

### Task 3 — `app` への状態・ユースケースの集約

- 共有符号化／復号、Undo／Redo 境界、プリセット適用、深度・予算・検証など、**ルールと編成**を `app` に寄せる。
- `ui` からの状態変更は、読み取りとイベント（またはアプリケーション層への明示的コマンド）に**統一されていく**ことを目標にする（一括ではなく本スライス範囲で許容する残差は次スライスで解消）。
- 本スライス完了時：ネイティブ・WASM ビルド成功、起動と主要編集のスモーク実施。
- [ ] 共有形式の詳細ロジックが `ui` だけに閉じ込められていない。

### Task 4 — `ui` の分割整理

- パネル・ビューポート・トースト等をサブモジュール化し、責務の見通しを揃える。
- `app` への依存は読み取りとイベントに限定し、[Phase 3 依存表](phase3_layout.md#モジュール間依存ルール許可関係)の **`ui` → `platform` 禁止**を満たす。
- 本スライス完了時：ビルド成功、UI 操作のスモーク（パラメータ・キャンバス・結果ビュー・通知表示）。
- [ ] `ui` がインフラ API を直接呼んでいない（ポート経由または `platform` が Bevy リソースとして提供する適合のみ）。

### Task 5 — `bootstrap`／エントリの薄型化

- プラグイン登録順・システムの束ねを `bootstrap`（または Phase 3 で許容した同等の場所）に閉じ、`main` は起動とターゲット別ブートストラップに薄く留める。
- 本スライス完了時：ビルド成功、起動スモーク。
- [ ] 起動処理だけが肥大化し、`core` への不必要な依存が増えていない（Phase 3 の `bootstrap` と `core` の注記と整合）。

### Task 6 — ファサード撤去と import の整理

- Task 1〜5 で入れた再エクスポートを削除し、呼び出し側の import を新ツリーへ寄せる。
- diff はレビュー可能な単位に分ける（**移動のみ**／**パス変更のみ**など、可能ならコミットを分離）。
- [ ] 旧パスへの依存が意図的に残っていない（一時的互換が必要な場合は例外と期限を本文またはコードコメントで明示）。

### Task 7 — 完了検証

- Phase 3 依存許可表に照らし、**禁止経路の `use` がない**ことを目視または簡易スクリプトで確認する（自動化は任意）。
- 手動スモークの最小範囲を実施し、結果を記録する（下記「受け入れ条件」）。
- [ ] [全体計画](overall_plan.md) の Phase 4（リファクタ完了時）受け入れをすべて満たす。

## 計画詳細

- **ポートの所在**：`ports` を独立させるか `app` に折りたたむかは Phase 3 の読み替えに従う。いずれにしても **`app` と `platform` の相互直接依存**は避ける（単一クレート内はモジュール順と分割で解決）。
- **`fractal_presets` 等の既存サブディレクトリ**：Phase 3 の論理領域に寄せる際、`core` との境界（データ vs 適用手続き）を崩さない。
- **非同期完了**：エクスポート・共有・トーストなど、Phase 2 で位置づけた「どの層が結果を発行するか」を、スライス後も一貫させる。挙動変更が必要になった場合は意図的な変更として記録する。
- **WASM**：`rfd` 等ネイティブ専用依存は引き続きターゲットで切り替え、ポート実装の差し替え先が `platform` に残ることを確認する。

## 受け入れ条件（検証）

### ビルド

- [ ] `cargo build`（リポジトリルート、デフォルトターゲット）が成功する。
- [ ] `trunk build --release` が CI（[deploy workflow](../../.github/workflows/deploy.yml)）と同旨の前提で成功する。

### 手動スモーク（最小）

リファクタ前と**同等**であること。項目は必要に応じて増減し、実施日と実行者を記録する。

- [ ] アプリが起動する（ネイティブ。WASM はビルドに加え、可能ならローカル表示で起動確認）。
- [ ] 種図形／配置の編集と Undo／Redo が破綻しない。
- [ ] プリセットまたはパラメータ変更が反映される。
- [ ] 共有（URL／クリップボード等、プロダクトが提供する経路）がエラーなく完走する。
- [ ] 画像エクスポートと通知（トースト等）が期待どおり動く。

### 構造・ルール

- [ ] [Phase 3 依存許可表](phase3_layout.md#モジュール間依存ルール許可関係)に明示的に反する依存経路がない。
- [ ] 意図的な一時ファサードが残る場合は、その理由と撤去条件が追跡できる。
