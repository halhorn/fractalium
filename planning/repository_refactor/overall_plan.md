# fractalium リポジトリリファクタ — 全体計画

## 目的

増築により責務が曖昧化し、ファイル・モジュールが肥大化している現状を是正する。挙動を保ちつつ、機能ドメインごとに境界を明確にし、将来の変更に耐える構造へ移行する。

## 提案フロー（合意事項）

次の 4 段階で進める。

1. **機能の洗い出しと責務の整理**  
   現状のファイル配置は参照せず、アプリが「何を提供するか」だけを列挙し、ドメイン／責務単位にグルーピングする。
2. **デザインパターンの選定**  
   fractalium（Bevy + UI、ネイティブ／WASM）に合うパターン・境界の付け方を文書化する。実装詳細や具体的な型名はこの段階では詰めすぎない。
3. **モジュール分割と配置**  
   上記責務に対応するモジュール階層・データの流れ・依存の向きを決め、ディレクトリ構成と主要クレート内の配置方針を確定する。
4. **実装**  
   段階的に移行し、各段階でビルド・主要動作を検証する。

この順序は妥当である。2 と 3 の間で「境界が固まったら配置を確定する」ことを明示しておくと、パターン議論が空振りしにくい。

## 計画概要

### Phase 1 — 機能一覧と責務マップ（ファイル非依存）

- アプリがユーザーに提供する能力を列挙する（例：フラクタル規則の選択・パラメータ変更、キャンバス操作、プリセット、状態の永続化・共有、画像エクスポート、トースト通知、プラットフォーム差の吸収など）。  
- 各能力を「コアドメイン」「アプリケーション」「インフラ／アダプタ」「プレゼンテーション」などの観点でタグ付けし、**まだ**既存 `src/*.rs` への対応は書かない（または付録に隔離する）。
- 成果物・Task 分解・受け入れ条件・機能一覧ドラフト：[phase1_capabilities.md](phase1_capabilities.md)。

### Phase 2 — パターンとアーキテクチャ方針

- Bevy の ECS・システム境界、egui との接続、WASM／ネイティブ分岐の扱いについて、採用する方針を短文で固定する（例：状態の単一ソース、UI は薄いアダプタ、I/O はtrait 背後に集約、など）。  
- 「採用する／採用しない」を理由つきでメモし、過剰な抽象化は避ける旨を受け入れ条件に含める。
- 成果物・Task 分解・受け入れ条件：[phase2_patterns.md](phase2_patterns.md)。

### Phase 3 — ディレクトリ構成と依存ルール

- Phase 1 の責務グループごとに `src/` 以下の候補ツリーを決める。  
- モジュール間の依存方向（例：ドメイン → インフラを直接参照しない）を箇条書きでルール化する。  
- 移行は「一括 big-bang」か「縦スライス（機能単位で旧→新）」かを決め、推奨は縦スライス＋既存公開 API の一時的なファサード。
- 成果物・Task 分解・受け入れ条件：[phase3_layout.md](phase3_layout.md)（ディレクトリ案、依存許可表、縦スライス順）。

### Phase 4 — 実装

- Phase 3 の順序に従い、モジュールの切り出し・型の移動・`mod` 再編を行う。  
- 各マイルストーン後に `cargo build`（およびプロジェクトで定めている WASM ビルド）と手動スモーク（起動、主要操作）を実施。  
- 成果物・Task 分解・受け入れ条件：[phase4_implementation.md](phase4_implementation.md)。リグレッション用の最小チェックリストは当該文書に集約する。

## 計画詳細

- Phase 1 の機能リストは、後続フェーズで「どこに置くか」が変わってもよいが、「ユーザーに見える能力」の漏れがないことが重要。  
- Phase 2 では一般的なクリーンアーキテクチャの四層をそのまま当てはめるより、Bevy らしい「リソース・イベント・システムの責務」に言い換えた方針の方が実装と整合しやすい。  
- Phase 4 は diff を小さく保つため、リネームのみ・移動のみのコミットを分けるとレビューしやすい。
