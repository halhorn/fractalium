# Agent instructions (fractalium)

このリポジトリでコードを触る AI / 人間開発者向けの共通指針。

## Rust の可読性・並び順

新規コード・リファクタ時に従う。既存ファイルを広範に並べ替えだけする変更は diff のノイズになるので、触る範囲で徐々に揃える。

### 全般方針
始めてコードを読む人がその内容をすばやく理解できるようする。

### 責務設計
プロダクトに求められている要件を抽象化し責務を考え、責務からモジュールやメソッドの設計を行う。
求められた新たな機能を実装するにあたり、責務のリファクタが必要な場合、必ず以下を添えて人間に許可を求める。
- 責務のリファクタを行うこと
- なぜ、どのような責務設計にするか
- 現状の設計との大まかな差分

### コメント
ユーザー価値を出すモジュールはユーザー価値始点で、システムのみのファイルは開発者価値視点でコメントを書く。
差分に対するコメントは不要。**はじめて**そのコードを読む人が理解しやすいようコメントを付ける。つまり、初めて読む人が疑問に思うであろう部分のみ経緯をコメントし、その他は経緯ではなく、現状のコードに対するコメントをする。
コメントの口調や使う単語は、一般的なコードのコメントの書式に習うこと。完結で理解しやすい文章を心がけること。

以下全てにコメントをつけること
- ファイル
- 定数
- struct
- メソッド、関数（引数、戻り値のコメント）

### 並び順

1. **`//!` モジュールドキュメント**（あれば）
2. **`use` / `extern` などインポート**
3. **モジュールの挙動を決める定数**（`pub` / 非公開は問わず。例：閾値、格子分割数、数学用の固定値）
4. **`pub` の型・列挙・trait・公開関数**（ファイルの「外から見た挙動」が上から追えるように）
5. **`pub(crate)` や、クレート内で再利用される実装詳細**
6. **非公開の補助関数・小さな型**（そのファイル／モジュール内部だけで完結）
7. **`#[cfg(test)] mod tests { ... }`** は原則 **ファイル末尾**

同一ブロック内では、「よりメイン・汎用」な `pub` をさらに上、特殊ケースを下。

### その他（可読性）

- **変更は依頼範囲に留める**：無関係な並べ替えやリネームだけの diff を避ける。
- **ドキュメント**：公開 API には `///` で「なぜ何をするか・引数／戻り」を書く。
- **可視性**：クレート外に不要なら `pub` にしない（`pub(crate)` やモジュール private）。
- **一貫性**：同じディレクトリや近いモジュールでは、既存ファイルの並び・ドコメント粒度に合わせる。
